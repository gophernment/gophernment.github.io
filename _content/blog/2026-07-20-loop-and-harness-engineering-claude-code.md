สวัสดีครับเพื่อน ๆ ชาวพัฒนาซอฟต์แวร์ทุกคน! มีใครเคยเจอปัญหานี้บ้างไหม? เวลาเราพยายามจะสร้าง LLM Agent หรือเขียนสคริปต์เพื่อครอบกระบวนการทำงานของบอต เช่น ให้มันช่วยแก้โค้ด รันเทสต์ แล้ววนแก้ใหม่... รอบที่หนึ่งก็ผ่านไปด้วยดี รอบที่สองก็ปกติดี แต่พอเข้ารอบที่สามปุ๊บ บอตเริ่มเอ๋อ เริ่มค้าง หรือไม่ก็วนลูปซ้ำ ๆ รันจนโทเคนบวม (Token blowup) เสียเงินไปฟรี ๆ โดยไม่ได้งาน!

เมื่อเร็ว ๆ นี้ผมไปสะดุดตาเข้ากับโพสต์หนึ่งบน X ของคุณ [@ArchiveExplorer](https://x.com/ArchiveExplorer/status/2071192832455430283) (เครดิตเต็ม ๆ ให้กับต้นทางเลยครับ ข้อมูลดีมาก ๆ จนผมต้องขอนำมาร้อยเรียงใหม่มาเล่าให้เพื่อน ๆ ฟัง) โพสต์นี้เขาชี้ทางสว่างไว้เลยครับว่า "ปัญหาของพวกเราส่วนใหญ่ไม่ใช่ที่ตัวลูป (Loop) แต่เป็นเพราะโครงสร้างพื้นฐานข้างใต้ที่ไม่ได้เตรียมไว้ต่างหาก" หรือที่เขาเรียกว่า **Harness** 

วันนี้เรามาคุยกันแบบเป็นกันเอง เจาะลึกเลเยอร์เหล่านี้กันดีกว่าครับว่าเครื่องเคราในโฟลเดอร์ `.claude/` ของ Claude Code มีอะไรที่เราต้องจัดระเบเบ้าง เพื่อให้เอเจนต์ของเราสามารถรันงานยาว ๆ ได้แบบไม่ตกหลุมพรางเดิม ๆ

## Harness vs Loop: สองเลเยอร์ที่ห้ามปนกัน (ห้องครัว กับ สูตรอาหาร)

ก่อนอื่นเราต้องแยกแยะให้ออกก่อนว่า ระบบเอเจนต์ที่ดีประกอบด้วย 2 เลเยอร์ที่แยกขาดจากกัน:

1. **Harness (โครงสร้างห้องครัว):** คือพวกโฟลเดอร์และไฟล์ตั้งค่าต่าง ๆ (เช่น ในโฟลเดอร์ `.claude/` ของเรา) เลเยอร์นี้จะไม่เปลี่ยนแปลงเลยในระหว่างการรันบอตแต่ละครั้ง มันคือขอบเขต กฎกติกา สิทธิ์การเข้าถึง และสเปกความสามารถที่บอตมี
2. **Loop (สูตรอาหาร):** คือกระบวนการที่รันอยู่ข้างบน Harness อีกที ตั้งแต่การรับเป้าหมาย (Goal) -> การลงมือทำ (Act) -> การตรวจสอบผลลัพธ์ (Verify) -> การจดจำความรู้ (Memory Write) -> ตัดสินใจว่าจะไปต่อหรือหยุดทำงาน

*ความเห็นของผมตรงนี้คือ:* เพื่อน ๆ หลายคน (รวมถึงตัวผมในอดีตด้วย) เวลาบอตรันแล้วค้าง หรือตอบมั่ว เรามักจะพุ่งตัวไปแก้ Prompt หรือพยายามเขียนเงื่อนไขควบคุมลูปก่อนเลย แต่ ArchiveExplorer บอกว่า **มันผิดเลเยอร์ครับ!** ถ้าห้องครัวไม่มีมีด ไม่มีเตาแก๊ส (ไม่มีสิทธิ์ Permission, ไม่มีไฟล์เช็กสถานะ) ต่อให้เขียนสูตรอาหารดีแค่ไหน ก็ทำอาหารออกมาไม่ได้ ปัญหาส่วนใหญ่ที่เราเจอ เช่น บอตค้างรอขออนุญาตเขียนไฟล์, บอตจำรายละเอียดเซสชันก่อนหน้าไม่ได้ ทั้งหมดนี้เป็นปัญหาของ **Harness** ทั้งสิ้น!

## เจาะลึก 7 ไฟล์หลักของ Harness ที่ต้องเซตให้ถูก

ลองเข้าไปดูในโฟลเดอร์ `.claude/` (หรือที่เกี่ยวข้อง) ของโปรเจกต์ระดับโปรที่ใช้ Claude Code คุณจะเจอ 7 สิ่งนี้ทำหน้าที่เป็นเลเยอร์พื้นฐานร่วมกัน:

### 1. `CLAUDE.md` — กฎเหล็กประจำบ้าน
ไฟล์นี้สำคัญที่สุด เพราะเป็นไฟล์แรกที่ Claude Code จะเข้าไปอ่านทุกครั้งเพื่อใช้เป็นบริบทพื้นฐาน (Standing Context) ในการทำงาน เราควรระบุโครงสร้างโปรเจกต์ เฟรมเวิร์กที่ใช้ คำสั่งรันเทสต์/บิวด์ที่ใช้งานได้จริง และสิ่งที่เรา "ห้ามบอตทำอย่างเด็ดขาด"

*ข้อควรระวัง:* อย่าเขียนเยอะเกินไปครับ! มีงานวิจัยชื่อ *Less Context, Better Agents* (arXiv 2606.10209) เผยตัวเลขที่น่าตกใจว่า เอเจนต์ที่ได้อ่านบริบทที่บวมเกินไป มีอัตราความสำเร็จในการทำงานลดลงจาก **91.6% เหลือเพียง 71%** ดังนั้น ขอแนะนำให้คอยหมั่นตรวจสุขภาพและล้างไฟล์นี้ทุกสัปดาห์ คุมความยาวให้อยู่ต่ำกว่า 300 บรรทัดก็พอครับ

### 2. `settings.json` — ด่านตรวจสิทธิ์
ไฟล์ที่กำหนด Allowlist ของคำสั่ง Bash หรือการเรียกใช้งาน MCP (Model Context Protocol) 

*ทริกเด็ดที่น่าลองคืนนี้:* ลองเพิ่มคำสั่งอ่านไฟล์พื้นฐาน เช่น `ls`, `git status`, หรือ `cat` ลงในรายการที่อนุญาตอัตโนมัติ (Auto-allow) เพื่อไม่ให้บอตหยุดชะงักถามหาคำอนุมัติจากเราทุก ๆ สองวินาที ส่วนคำสั่งแก้ไขหรือลบโค้ดยังคงต้องปล่อยให้มีหน้าต่างยืนยันความปลอดภัยไว้ครับ

### 3. `hooks` — สคริปต์อัตโนมัติที่ช่วยล็อกฟอร์แมต
ช่วยควบคุมพฤติกรรมหลังการใช้งานเครื่องมือต่าง ๆ เช่น ตั้งค่า `PostToolUse` ให้สแกนไฟล์ที่ถูกแก้ไข แล้วสั่งรันล้างโค้ดด้วย Prettier เสมอ เพื่อให้มั่นใจว่าโค้ดที่ผลิตออกมาอยู่ในรูปแบบที่ระบบตรวจจับได้ทันที

### 4. `agents` — ผู้ตรวจการแยกอิสระ
อยู่ภายใต้โฟลเดอร์ `.claude/agents/` เป็น Subagent ที่เราเรียกใช้งานผ่านเครื่องมือส่งงาน (Task tool) 

*คอมเมนต์สไตล์นักพัฒนาร่วมสายงาน:* ตัวบอตเขียนโค้ดมักจะเข้าข้างตัวเองครับ (ฮา) ดังนั้นการให้บอตคนเดิมมาตรวจงานตัวเอง มักจะลงเอยด้วยคำว่า "ผ่านฉลุย" ทั้งที่พังยับเยิน การสร้าง Subagent ที่มีระบบ Context สะอาดและโฟกัสกับการตรวจทานโดยเฉพาะ จึงเป็นท่าที่ลดความเสี่ยงโค้ดขยะได้มากที่สุด

### 5. `skills` — คลังวิชาเฉพาะทาง
โฟลเดอร์เก็บความสามารถเสริมของบอต เช่น การแก้ไขไฟล์ตามรูปแบบเฉพาะ เราเก็บเป็นหัวข้อย่อยเพื่อที่จะโหลดเข้าบริบทของ Claude ก็ต่อเมื่อจำเป็นเท่านั้น ไม่ได้ยัดส่งไปแต่แรก ช่วยประหยัด Token ได้มหาศาลครับ

### 6. `.mcp.json` — สะพานเชื่อมโลกภายนอก
การตั้งค่า Model Context Protocol เพื่อเชื่อมบอตเข้ากับระบบภายนอก เช่น การดึงคอมมิตใน GitHub หรืออ่านเอกสาร API สด ๆ ข้อพึงระวังคืออย่าเปิดสิทธิ์เขียน (Write scope) ให้มั่วซั่วโดยไม่มีระบบ Log คอยเฝ้าดูเด็ดขาดครับ

### 7. `MEMORY.md` & `vault` — สมุดบันทึกความจำระยะยาว
เป็นไฟล์เก็บความรู้ข้ามเซสชัน (State & Memory) ช่วยบันทึกสิ่งที่บอตเรียนรู้ไปแล้วว่าวิธีไหนลองแล้วไม่ได้ผล หรือความชอบส่วนตัวของเรา เพื่อให้บอตไม่ต้องเริ่มนับหนึ่งใหม่ในวันถัดไป แต่จำไว้ว่าเราต้องหมั่นเคลียร์มันด้วยนะครับ ไม่ใช่ปล่อยให้มันโตเป็นขยะความจำ (Context rot) จนบอตสับสน

## วงโคจร 5 ขั้นตอนของ Loop สู่ความสำเร็จ

หลังจากที่ Harness ของเรามั่นคงแล้ว ลูปการทำงานบนพื้นฐานนั้นจะมีกระบวนการ 5 ส่วนดังนี้:

1. **Goal Spec:** เป้าหมายที่บันทึกไว้ในดิสก์ เช่น `PROMPT.md` หรือ `AGENT_SPEC.md` ตัวบอตจะกลับมาอ่านทุกรอบเพื่อป้องกันปัญหาระดับคลาสสิกอย่าง "เดินออกนอกเส้นทาง"
2. **Plan-Act-Verify:** การวางแผน ลงมือทำ และตรวจสอบด้วยระบบแยก Context เพื่อป้องการความมั่นใจในข้อมูลที่ผิดพลาด (Confident Garbage)
3. **Sub-agent Fan-out:** เมื่อเป็นงานขนาดใหญ่ ให้ใช้วิธีแตกงานออกเป็นส่วนย่อย ๆ ให้เอเจนต์ขนาดเล็กทำงานคู่ขนานกัน แทนที่จะพยายามใช้เอเจนต์ตัวเดียวทำทุกอย่างในบริบทบวม ๆ 
4. **Scheduler & Persistence:** ตัวสั่งการให้ลูปรันตอนที่เราไม่อยู่หน้าจอ เช่น ใช้ `cron` หรือ `launchd` บน macOS ร่วมกับการบันทึกประวัติการทำงานในรูปแบบดิสก์ เพื่อให้เวลาที่บอตถูกปลุกขึ้นมารันใหม่ มันยังรู้ตัวว่าต้องทำงานอะไรต่อ
5. **การรู้เท่าทัน 3 จุดตาย (Failure Modes):**
   * *Confident garbage:* ตรวจสอบผลลัพธ์ผ่าน ๆ แล้วผ่านเลย ทำให้โค้ดเน่า ๆ หลุดเข้ามา
   * *Context rot:* การสะสมประวัติแชตยาวเกิน 200,000 โทเคน จนโมเดลเริ่มเบลอ
   * *Ralph Wiggum loops:* ลูปวนซ้ำเพราะไม่มีการอัปเดตสถานะความคืบหน้าลงบนดิสก์ บอตจึงมาเริ่มคิดแผนใหม่และทำซ้ำขั้นตอนเดิมที่เคยทำเสร็จแล้ว

## ข้อเตือนใจส่งท้าย

> [!NOTE]
> เทคโนโลยีการพัฒนา AI Agent และฟีเจอร์ต่าง ๆ ของ Claude Code กำลังพัฒนาไปอย่างรวดเร็วมาก (ข้อมูลนี้อ้างอิงตามสถาปัตยกรรมและเครื่องมือ ณ ช่วงกลางปี 2026) แนวคิดของ Harness & Loop จึงอาจมีการเปลี่ยนแปลงเชิงลึกได้อีกในอนาคต ดังนั้นเราควรทดลองนำไปใช้อย่างยืดหยุ่นและคอยติดตามอัปเดตใหม่ ๆ เสมอครับ

คืนนี้ลองเปิดโฟลเดอร์ของเพื่อน ๆ ดูนะครับ ถ้ายังเห็นแค่โฟลเดอร์ว่างเปล่า หรือมีแค่ `settings.json` ลองเริ่มจากการเขียน `CLAUDE.md` สั้น ๆ ไม่เกิน 300 บรรทัดดูก่อน แล้วค่อย ๆ ประกอบร่าง Harness ของตัวเองทีละชิ้น ขอให้สนุกกับการสร้างลูปที่ไม่ล่มครับ!
